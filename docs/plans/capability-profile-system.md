# Capability Profile System — Full Plan

## Overview

Replace the hand-maintained capability boilerplate with a code-generated system driven by a single TOML file. The TOML is the single source of truth for all qBittorrent capability definitions. A Node codegen script (`scripts/codegen/capabilities.mjs`) reads the TOML and emits Rust structs + TypeScript interfaces. Generated files are gitignored and regenerated by common pnpm dev/build/test/CI scripts before they are needed. `build.rs` performs read-only validation of the TOML — it does not write any files.

## Key Decisions (from grill-me session)

| Decision | Choice |
|----------|--------|
| Capability scope | Fine-grained (~20 caps): sub-features like `supports_rss`, `supports_rss_refresh`, `supports_rss_clone`, etc. |
| Version history | Sparse TOML: only versions where capabilities actually changed |
| Removals | TOML supports `adds` and `removes` per version |
| Lifecycle metadata | **Not emitted into generated code.** TOML is the lifecycle documentation (version-ordered, with `adds`/`removes` per version). Generated code emits only the runtime contract: struct, `KNOWN_CAPABILITIES`, `set_capability`. If a debug UI is needed later, it reads the embedded TOML directly. |
| Naming convention | Full capability name (`supports_x`) = the generated struct field name |
| Codegen location | **Node script** (`scripts/codegen/capabilities.mjs`) using `@iarna/toml`. Single writer for Rust + TS output. `build.rs` is validation-only. |
| Generated files | **Gitignored.** Common pnpm scripts and CI generate them before TypeScript/Rust checks to avoid mechanical merge conflicts in generated source. |
| Codegen output | Rust `generated.rs` + TS `server-capabilities.ts` (bridge) + `app-capabilities.ts` (web-core). All emit `// @generated. DO NOT EDIT.` headers. |
| TS output path | Generated at final source paths: `packages/bridge/src/generated/`, `packages/web-core/src/capabilities/generated/` |
| Version corrections | `[corrections]` section in TOML: `"2.7.0" = "2.8.0"` (exact-match only) |
| Source data | qBittorrent wiki API pages + source code git blame archaeology for unverifiable boundaries |
| Archaeology | Complete before shipping: every capability must have verified `added_in`/`removed_in` |
| Migration | Delete-and-generate: remove manual files, codegen overwrites |
| `supports_pause_resume` | Moved to `[app_versions]` TOML section (keyed on qBittorrent app version, not webapi version) with `removes` at the v5 boundary |
| `[base]` section | **Removed.** Everything lives in `[versions."2.0"]`. v2.0-floor assumption: no qBittorrent instance reports webapi below 2.0 in practice. |
| Test strategy | **Generated test tables + mock helpers.** Codegen emits parameterized `(version, expected_caps)` table for Rust resolver tests, and `makeServerCapabilities()` / `makeAppCapabilities()` helpers for TS tests. All tests switch from hardcoded literals. |
| Scope boundary | Server capabilities only (`ResolvedCapabilities`/`ServerCapabilities`/`AppCapabilities`). `BridgeCapabilities` (platform-level) is orthogonal and unchanged. |

## 1. TOML as Single Source of Truth

File: `crates/qb-core/capabilities/qbittorrent-capabilities.toml`

### Shape

```toml
# Known upstream webapiVersion bugs → actual version (exact-match only)
[corrections]
"2.7.0" = "2.8.0"

# ── WebAPI-version-keyed capabilities ───────────────────────────────
# Resolved via: GET /api/v2/app/webapiVersion
# v2.0-floor assumption: no qBittorrent instance reports below 2.0.
# Sparse: only versions where capabilities actually changed.

[versions."2.0"]
app_version = "v4.1.0"
adds = [
  { name = "supports_rss", description = "RSS feed API (/api/v2/rss/*)" },
]

[versions."2.1.1"]
app_version = "v4.1.4"
adds = [
  { name = "supports_search", description = "Search API (/api/v2/search/*)" },
]

[versions."2.2.0"]
app_version = "v4.1.5"
adds = [
  { name = "supports_tracker_editing", description = "Tracker edit/remove endpoints" },
]

[versions."2.2.1"]
app_version = "v4.1.9"
adds = [
  { name = "supports_rss_refresh", description = "RSS feed refresh endpoint" },
]

[versions."2.3.0"]
app_version = "v4.2.0"
adds = [
  { name = "supports_tags", description = "Tags management API" },
]

[versions."2.4.0"]
app_version = "v4.2.1"
adds = [
  { name = "supports_file_renaming", description = "Torrent file renaming" },
]

[versions."2.8.0"]
app_version = "v4.3.3"
adds = [
  { name = "supports_folder_renaming", description = "Torrent folder renaming" },
]

# ── App-version-keyed capabilities ──────────────────────────────────
# Resolved via: GET /api/v2/app/version
# Used for capabilities whose lifecycle is tied to the app release version,
# not the webapi version. Currently only supports_pause_resume.

[app_versions."v4.1.0"]
adds = [
  { name = "supports_pause_resume", description = "Torrent pause/resume endpoints" },
]

[app_versions."v5.0.0"]
removes = [
  { name = "supports_pause_resume", description = "Pause/resume removed in qBittorrent v5.0" },
]

# ── Capabilities requiring archaeology for exact boundaries ─────────
#   supports_rss_rules          - RSS auto-download rule CRUD
#   supports_rss_matching       - RSS matching articles endpoint
#   supports_rss_clone          - RSS clone rule endpoint
#   supports_webseed_management - Webseed add/edit/remove endpoints
#   supports_categories_manage  - Category CRUD with savePath
#   supports_metadata_api       - fetchMetadata, parseMetadata, saveMetadata
#   supports_torrent_comments   - setComment endpoint
#   supports_api_key_auth       - API key authentication
#   supports_basic_auth         - Basic auth support
#   supports_speed_limits_api   - getSpeedLimits, setSpeedLimits
#   supports_file_download      - downloadFile endpoint
#   supports_piece_availability - pieceAvailability endpoint
#   supports_process_info       - processInfo endpoint
```

### Design rules
- **Sparse**: only versions where capabilities actually changed (not 40+ entries)
- **`adds`** entries define capability name + description per version
- **`removes`** entries reference capability names (must have been previously `adds`-ed and not already removed)
- **`[versions]`** is keyed on webapi version (`GET /api/v2/app/webapiVersion`)
- **`[app_versions]`** is keyed on app version (`GET /api/v2/app/version`) for capabilities whose lifecycle follows app releases
- **`[corrections]`** handles known upstream webapiVersion bugs (exact-match only)
- **`app_version`** field within each version entry is metadata for docs/observability, not used by the resolver
- **No `[base]`** section: the v2.0 floor is explicit in `[versions."2.0"]`
- **No lifecycle metadata** in generated code: the TOML *is* the lifecycle record
- **TOML is the canonical capability name list**: validation verifies `removes` references an earlier `adds`, not a separate hardcoded list

## 2. Codegen Architecture

### 2a. Node Codegen Script (`scripts/codegen/capabilities.mjs`)

**Single writer for all 3 generated files.** Reads the TOML, validates invariants, emits:

| Output | Path | Contents |
|--------|------|----------|
| Rust | `crates/qb-core/src/capability/generated.rs` | `ResolvedCapabilities` struct, `KNOWN_CAPABILITIES`, `set_capability()`, and the `RESOLVER_TEST_TABLE` for parameterized tests |
| TS (bridge) | `packages/bridge/src/generated/server-capabilities.ts` | `ServerCapabilities` interface, `makeServerCapabilities()` helper |
| TS (web-core) | `packages/web-core/src/capabilities/generated/app-capabilities.ts` | `AppCapabilities` interface, `DEFAULT_APP_CAPABILITIES`, `toAppCapabilities()`, `makeAppCapabilities()` helper |

All emitted files include a `// @generated. DO NOT EDIT.` header.

The script supports `--check` mode: regenerates to a temp dir, diffs against current generated outputs, exits non-zero on drift. CI uses normal generation instead because generated outputs are intentionally untracked.

### 2b. `build.rs` (Validation-Only)

`build.rs` parses the TOML and validates structural invariants. It writes **no files**:

1. All version keys (`[versions]` and `[app_versions]`) are valid semver
2. Every capability in `removes` was previously in `adds` (ordered traversal, not a name lookup list)
3. Every capability in `removes` hasn't already been removed by an earlier version
4. `[corrections]` keys and values are valid semver

No cross-workspace paths. No TS output. No race conditions with parallel cargo invocations.

### 2c. Generation Wiring

- **`build.rs`**: catches TOML structural errors during any cargo invocation
- **`pnpm codegen:capabilities`**: regenerates Rust/TypeScript outputs from TOML
- **`pnpm install`**: postinstall runs codegen so clean checkouts have generated files after dependency install
- **Common pnpm dev/build/test scripts**: run codegen before invoking Vite, Tauri, Vitest, Playwright, TypeScript, or Cargo checks
- **GitHub Actions**: run codegen before JS quality, Rust quality, native smoke, and release build steps
- **`pnpm codegen:check`**: remains available for local drift/debug checks against current generated outputs

### 2d. New Dependencies

| Dependency | Location | Purpose |
|------------|----------|---------|
| `@iarna/toml` | Root `package.json` devDependencies | TOML parsing in Node codegen |

### 2e. New Scripts (root `package.json`)

```json
{
  "scripts": {
    "codegen:capabilities": "node scripts/codegen/capabilities.mjs",
    "codegen:check": "node scripts/codegen/capabilities.mjs --check"
  }
}
```

### 2f. Generated Code Formatting

- **Rust**: codegen emits `rustfmt`-compliant output (or pipes through `rustfmt`), includes `// @generated` header
- **TypeScript**: codegen emits prettier-compliant output (or pipes through prettier), includes `// @generated` header
- Generated directories added to `.eslintignore` / `.prettierignore` patterns if needed (or just emit pre-formatted)

## 3. Resolution Algorithm

Two-pass: webapi-version caps first, then app-version caps.

```
resolve(webapi_version_str, app_version_str):
  1. Apply [corrections] to webapi_version_str
  2. Parse corrected webapi_version as (major, minor, patch)
  3. Parse app_version_str as (major, minor, patch)
  4. caps = all-false ResolvedCapabilities

  # Pass 1: webapi-version-keyed capabilities
  5. For each [versions."X.Y.Z"] where X.Y.Z <= webapi_version:
     a. For each cap in adds: set_capability(&mut caps, cap, true)
     b. For each cap in removes: set_capability(&mut caps, cap, false)

  # Pass 2: app-version-keyed capabilities
  6. For each [app_versions."X.Y.Z"] where X.Y.Z <= app_version:
     a. For each cap in adds: set_capability(&mut caps, cap, true)
     b. For each cap in removes: set_capability(&mut caps, cap, false)

  7. Return caps
```

## 5. Testing

### 5a. Codegen unit tests (`scripts/codegen/capabilities.test.mjs`)

- Parse valid TOML → correct generated output for all 3 files
- Parse invalid TOML → validation errors
- `removes` referencing non-existent cap → error
- `removes` referencing already-removed cap → error
- Invalid semver keys → error
- `--check` mode detects drift

### 5b. Generated resolver tests

The codegen emits a `RESOLVER_TEST_TABLE` into `generated.rs`:

```rust
// Auto-generated by scripts/codegen/capabilities.mjs. DO NOT EDIT.
pub const RESOLVER_TEST_TABLE: &[(&str, &[(&str, bool)])] = &[
    ("2.0.0", &[
        ("supports_rss", true),
        ("supports_search", false),
        // ...
    ]),
    ("2.1.1", &[
        ("supports_rss", true),
        ("supports_search", true),
        // ...
    ]),
    // ... one entry per version boundary
];
```

The resolver's existing test suite switches to a single parameterized test that iterates this table and verifies `resolve()` produces the expected booleans. This ensures tests stay in sync with the TOML automatically.

### 5c. Generated TS mock helpers

`makeServerCapabilities(overrides?: Partial<ServerCapabilities>)` — returns a full `ServerCapabilities` with all fields set to `false`, applying any overrides. All TS tests switch from hardcoded partial literals to this helper, so adding a capability doesn't break every test.

`makeAppCapabilities(overrides?: Partial<AppCapabilities>)` — same for the camelCase version.
