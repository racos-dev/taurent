# capabilities

## Responsibility

Server capability consumption from the Rust backend. Exposes the `ServerCapabilities` payload (produced by `qb-core::capability::QbResolver`) into a typed `AppCapabilities` shape consumed by the UI layer. Rust is the single source of truth for capability flags — there is no TypeScript probe or version-resolution fallback.

## Key Files

- `index.ts` — Barrel re-exporting from `./generated/app-capabilities`.
- `generated/app-capabilities.ts` — Auto-generated (`@generated by scripts/codegen/capabilities.mjs`). Defines the `AppCapabilities` interface (21 camelCase boolean fields), the `DEFAULT_APP_CAPABILITIES` constant (all `false`), the `toAppCapabilities()` mapper that converts a `ServerCapabilities` (snake_case, imported from `@taurent/bridge`) into `AppCapabilities`, and the `makeAppCapabilities()` factory for test overrides. Source of truth: `crates/qb-core/capabilities/qbittorrent-capabilities.toml`.

## Design Patterns

- **Rust as source of truth**: All capability flags are derived from Rust's `ServerCapabilities`, produced by `QbResolver::resolve` and shipped to the renderer through `SessionSnapshot.capabilities` on every session transition. The TypeScript side does no version parsing, no probe calls, and no fallback resolution.
- **Wire format translation**: `ServerCapabilities` uses snake_case (matches Rust `#[serde(rename_all = "snake_case")]`); `AppCapabilities` uses camelCase to match the rest of the renderer. `toAppCapabilities()` performs the 1-to-1 field rename.
- **Boolean, not tri-state**: The current `AppCapabilities` surface is plain `boolean` — every flag is either `true` or `false`. There is no `null`/`unknown` middle state; the Rust resolver defaults to all-false on parse failure and the mapper preserves that. Consumers can treat every field as a feature gate.

## Flow

1. The Rust `QbResolver::resolve(webapi_version, app_version)` produces a `ResolvedCapabilities` after a successful login and stores it on `SessionState`.
2. Every `SessionSnapshot` carries the snake_case `ServerCapabilities` shape (mirrored from the Rust struct) under `capabilities`.
3. `useStandardContextValue` maps it via `toAppCapabilities()` into the camelCase `AppCapabilities` and exposes it on `QBClientContextValue.capabilities`.
4. Feature controllers (`useRssController`, `useSearchController`, torrent action controllers) read the relevant flags directly as feature gates.

## Integration

- Exports `AppCapabilities` type and `DEFAULT_APP_CAPABILITIES` / `toAppCapabilities` / `makeAppCapabilities` helpers.
- `ServerCapabilities` is the bridge-owned snake_case wire shape, imported from `@taurent/bridge` (itself re-exported from `./generated/server-capabilities`).
- Used by `rss/useRssController` (gates RSS UI on `capabilities.supportsRss`), `search/useSearchController` (gates search UI on `capabilities.supportsSearch`), and torrent action controllers (`supportsPauseResume`, `supportsSpeedLimitsApi`, etc.).
- Consumed by `useStandardContextValue` which performs the snake_case → camelCase mapping once per session transition.
