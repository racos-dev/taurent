# Repository Atlas: Taurent

## Project Responsibility

Taurent is a pnpm monorepo for qBittorrent clients built around a shared frontend architecture. Desktop and mobile apps stay thin while shared packages own transport abstraction, validated domain types, session/query orchestration, and reusable UI.

## System Architecture

- **Apps as shells**: `apps/desktop` and `apps/mobile` compose providers, routing, and platform UX.
- **Bridge boundary**: `packages/bridge` owns platform transport, typed bridge contracts, and desktop/mobile adapters.
- **Headless core**: `packages/web-core` owns session lifecycle, query scoping, hooks, and screen controllers/models.
- **Shared domain layer**: `packages/shared` owns schemas, types, utilities, theme tokens, and stores.
- **UI layer**: `packages/web-ui` owns primitives, dialogs, settings/torrent/management components, and screen bodies.
- **Rust/native layer**: `crates/qb-core` owns the platform-agnostic qBittorrent client (HTTP, auth, session, domain types). `crates/qb-tauri` owns the Tauri bridge surface — managed state, server repository, commands, session event emission, and OS-level concerns. App Tauri crates (`apps/desktop/src-tauri`, `apps/mobile/src-tauri`) are thin shells that compose plugins, register commands, and own platform-specific UI integrations (tray, menu, notifications for desktop; sidecar structure for mobile).

## Current Structural Shape

- `packages/bridge/src/` is now split into **`contracts/` → `transport/` → `adapters/`**.
- `packages/web-core/src/session/` replaces the older connection-centric layout for session/provider wiring.
- `packages/web-core/src/screens/` now holds feature controllers/models that compose lower-level hooks.
- `packages/web-ui/src/components/` is reorganized by domain: `primitives`, `layout`, `dialogs`, `management`, `server-setup`, `settings`, `shared`, `torrents`.
- `apps/desktop/src/hooks/` is reorganized into `platform/`, `settings/`, `shell/`, and `torrents/`. Live torrent hooks (`useLiveTorrentList`, `useLiveTorrentByHash`, `useLiveTorrentsByHash`) subscribe to the torrents map slice via `useMaindataSelector` and derive via `useMemo` to avoid full Zustand array subscriptions on unrelated maindata ticks; `useLiveTorrentList` includes an optional identity-churn probe gated by `isPerfAuditEnabled()`.
- `apps/desktop/src/windows/` now centers on route-driven auxiliary windows and dedicated dialog windows rather than in-tree desktop overlays.

## Runtime Flow

1. App entrypoints create a single QueryClient, then compose theme, server-manager, and session providers.
2. `web-core` session factories consume bridge adapters and expose scoped hooks/context values.
3. Hooks and screen controllers call bridge methods through typed contracts.
4. Bridge adapters map calls onto Tauri transport commands/listeners.
5. Rust/Tauri handlers talk to qBittorrent. All DTO families (torrent list, properties, trackers, files, categories, tags, search, RSS, peers) are parsed and validated by `qb-core::dto` inside the Tauri command and returned as typed envelopes. The live-sync path is backend-owned: `qb-tauri` sync managers poll and accumulate deltas, emit `maindata-sync-changed` events with optional embedded raw deltas for the fast path, and serve full snapshots on demand for the slow path.

## Entry Points and Read Order

1. `crates/qb-core/src/lib.rs` — platform-agnostic qBittorrent client surface: client functions, error types, session manager, server domain DTOs
2. `crates/qb-tauri/src/lib.rs` — Tauri bridge: managed state handles, command exports, session/error types, app builder plugin helpers
3. `packages/bridge/src/contracts/interfaces.ts` — canonical frontend-to-platform contract
4. `packages/web-core/src/session/` — session bootstrap, context values, invalidation, lifecycle
5. `packages/web-core/src/screens/` — screen controllers/models built from hooks
6. `packages/web-ui/src/index.ts` and `src/components/` — grouped UI export surface
7. `apps/desktop/src/App.tsx` and `apps/mobile/src/App.tsx` — final provider/routing composition

## Key Constraints

- Rust owns application, integration, domain, filesystem, network, qBittorrent, validation, normalization, synchronization, persistence, and native-platform logic that Rust can reasonably handle. TypeScript/React owns frontend UI concerns: rendering, component state that is purely visual, route composition, user interactions, and presentation-specific view models.
- `packages/web-core`, `packages/web-ui`, and `packages/shared` must not import `@tauri-apps/*`.
- Desktop auxiliary windows must reuse the shared session snapshot; they must not create independent client sessions.
- New shared behavior should default into `packages/shared`, `packages/web-core`, or `packages/web-ui` before app-specific placement.
